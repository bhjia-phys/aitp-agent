# AITP Agent 0.0.1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the first physics-native AITP Agent vertical slice: a skill-like, progressively disclosed physics memory system with a lightweight compiler that turns LibRPA knowledge into typed capsules, context packs, and validation records.

**Architecture:** Extend `packages/agent-core` with a parallel `physics-memory` subsystem rather than putting physics knowledge directly into skills. Skills remain procedural memory; physics capsules become typed semantic memory; the compiler validates and assembles domain-scoped context packs. The first release should prove the shape with one LibRPA formula-to-code/debug vertical slice.

**Tech Stack:** TypeScript, Kimi Code `agent-core`, existing `SkillRegistry`/`ToolManager`/`AgentRecords`, Vitest, JSON Schema style validation through local TypeScript types for 0.0.1.

---

## Product thesis

AITP Agent should not be an external research notebook bolted onto a coding agent. It should make theoretical-physics memory a first-class runtime substrate:

- The model starts with a compact index, not a large protocol prompt.
- The model can actively load relevant theory capsules the same way it loads skills.
- Domain contexts stay isolated unless an explicit bridge capsule allows crossing.
- New derivations, code failures, and benchmark results can be compiled back into typed memory.
- Reliability is represented as state: raw, parsed, linked, checked, validated, or formalized.

For 0.0.1, do not attempt a full physics brain. Build the narrow but real path:

```text
LibRPA raw/domain notes
-> typed physics memory capsules
-> compiler checks and context packs
-> model-loadable physics memory tool
-> validation/failure records
```

## Repository decision

Use `F:/AI_Workspace/repos/aitp-agent` as the primary implementation base.

The deciding point is not that ForgeCode lacks skills. ForgeCode also has progressive skill disclosure through an available skills list and a `skill` tool. The difference is runtime semantics:

- Kimi Code records skill activation and injects loaded skill content as a session context event.
- ForgeCode fetches skill details as a normal tool result.
- Kimi Code already has `SkillRegistry`, model-invocable `Skill`, session hooks, MCP, subagents, records, replay, and compaction in `packages/agent-core`.
- ForgeCode should be mined for evaluation/harness ideas, especially YAML/CSV evals, tool boundary strictness, and explicit agent definitions.

The 0.0.1 implementation should therefore follow Kimi Code's runtime shape and borrow ForgeCode's benchmark style later.

## Imported design constraints from Codex and ForgeCode

0.0.1 should not reimplement Codex or ForgeCode, but it should reserve the interfaces needed to absorb their useful ideas.

### Codex-inspired tool/action lifecycle

Codex has a clean split between tool spec, exposure, invocation source, structured output, and pre/post hook payloads. AITP Agent should carry those ideas into `research-action` without rewriting Kimi's loop:

- `ActionExposure`: `direct`, `deferred`, `direct-model-only`, `hidden`.
- `ResearchActionSource`: `model`, `controller`, `hidden-check`, `subagent`, `replay`.
- stable pre-action and post-action payloads that include action id, input, graph refs, capsule refs, outcome, and evidence refs.
- structured outputs suitable for both model context and harness scoring.

For 0.0.1, implement only the foundation types and registry. Do not yet replace Kimi's tool manager.

### ForgeCode-inspired harness boundary

ForgeCode's benchmark directory uses task definitions, data sources, validations, timeouts, and debug artifacts. AITP Agent should mirror that shape for physics research without adopting the whole ForgeCode runner:

- `ResearchEvalCase`: task description, action sequence, expected graph/capsule effects, validations, and evidence expectations.
- `ResearchActionRecord`: append-only action traces that can be converted into eval cases.
- validation types should be general enough for regex/shell-like checks later, but 0.0.1 should keep them as typed data only.

This keeps the self-evolving harness idea connected to runtime traces from the beginning.

## Conceptual boundaries

Do not collapse these into one concept.

### Skills

Skills are procedural memory. They answer: "How should the agent work?"

Examples:

- `librpa-formula-to-code-debug`
- `derive-then-check-dimensions`
- `compile-new-physics-note`
- `build-benchmark-from-failure`

### Physics memory capsules

Capsules are typed semantic memory. They answer: "What is known, under which assumptions, with which dependencies and evidence?"

Examples:

- `Definition`
- `Formula`
- `Assumption`
- `DerivationStep`
- `CodeMapping`
- `IntermediateObservable`
- `BenchmarkCase`
- `FailureMode`
- `WorkflowRecipe`

Capsules are not the whole graph. They are compiled projections of a finer physics graph. Each capsule should carry:

- `graphRefs`: links to graph atoms such as formulas, derivation steps, code mappings, benchmarks, or failure modes.
- `expansionHandles`: stable handles for loading deeper graph detail.
- `requiredChecks`: check contracts that should run before the capsule is trusted in a derivation or code change.
- `actionAffordances`: research actions that are allowed, recommended, or required when the capsule is active.

### Compiler

The compiler is neither a skill nor memory. It is a checker and context-pack builder:

```text
raw source
-> parsed capsule
-> linked dependency graph
-> consistency checks
-> domain context pack
-> validation/failure feedback
```

### Context pack

The context pack is what the agent actually sees during a task. It should be small and domain-scoped:

- active domain
- capsule index
- selected capsule summaries
- applicable workflow skills
- compiler warnings
- validation status
- unresolved contradictions

## 0.0.1 scope

### In scope

- Add a `physics-memory` package area under `packages/agent-core/src/physics-memory`.
- Define capsule metadata and typed capsule body shapes.
- Include graph references, expansion handles, required checks, and action affordances in capsule metadata.
- Discover physics memory roots from project and user locations.
- Provide a registry similar to `SkillRegistry`.
- Add a model-invocable `PhysicsMemory` builtin tool.
- Add a simple compiler that builds a LibRPA context pack from local fixture capsules.
- Record load/compile events through `AgentRecords`.
- Add tests for discovery, parsing, registry lookup, context pack compilation, and tool execution.
- Add one LibRPA fixture capsule set for formula-to-code/debug planning.
- Add a foundation-only `research-action` package area with Codex-inspired exposure/source types and ForgeCode-inspired action/eval record types.

### Out of scope

- Full AITP MCP integration.
- Lean integration.
- Real symbolic algebra.
- Full graph database.
- Vector search/RAG.
- Web UI.
- Automatic promotion to trusted memory.
- Cross-domain theory morphisms beyond a placeholder bridge type.
- Full action execution over shell/git/web/remote compute.
- Full ForgeCode-style eval runner.

## Capsule schema draft

Use Markdown with YAML frontmatter for 0.0.1. This mirrors skills while keeping semantics separate.

Example:

```markdown
---
id: formula.librpa.chi0.independent-particle
kind: Formula
domain: librpa
title: Independent-particle polarizability
symbols:
  - chi0
  - omega
  - q
assumes:
  - assumption.rpa.no-vertex-correction
depends_on:
  - definition.librpa.single-particle-state
reliability: linked
source_refs:
  - local:librpa-notes/chi0.md
graph_refs:
  - kind: Formula
    id: graph.librpa.formula.chi0
expansion_handles:
  - kind: derivation
    ref: graph.librpa.derivation.chi0
required_checks:
  - id: check.dimension.chi0
    kind: dimension
    severity: warning
action_affordances:
  - action_id: derive.check_dimension_consistency
    intent: recommended
  - action_id: code.map_formula_to_code_region
    intent: recommended
---

## Statement

The independent-particle polarizability used by LibRPA is represented as ...

## Checks

- dimension: pending
- symmetry: pending
- benchmark: pending
```

Reliability states:

- `raw`
- `parsed`
- `linked`
- `checked`
- `validated`
- `formalized`
- `rejected`

## Tool shape

The `PhysicsMemory` tool should support four actions in 0.0.1:

```ts
type PhysicsMemoryInput =
  | { action: 'list_domains' }
  | { action: 'list_capsules'; domain?: string; kind?: string }
  | { action: 'load_capsule'; id: string }
  | { action: 'compile_context'; domain: string; focus?: string[] };
```

Tool outputs should be compact XML-like text, following existing Kimi tool-result style:

```xml
<physics_memory_context domain="librpa" reliability_floor="linked">
  <capsule id="formula.librpa.chi0.independent-particle" kind="Formula" reliability="linked">
    ...
  </capsule>
  <diagnostic severity="warning" code="missing-benchmark">
    Formula has no validated benchmark case.
  </diagnostic>
</physics_memory_context>
```
