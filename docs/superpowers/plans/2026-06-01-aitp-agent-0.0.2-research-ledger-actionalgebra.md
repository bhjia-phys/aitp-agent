# AITP Agent 0.0.2 Research Ledger And ActionAlgebra Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the runtime layer that turns real research sessions into compile-ready evidence and typed, auditable research actions.

**Architecture:** Keep the 0.0.1 `physics-memory` layer as compiled semantic memory. Add a parallel `research-ledger` layer for source-backed research events, then extend `research-action` into an ActionAlgebra that creates obligations, schedules checks, and coordinates existing Kimi/Codex-style primitive tools without replacing them.

**Tech Stack:** TypeScript, Kimi Code `agent-core`, existing `ToolManager`/`AgentRecords`/`Session` lifecycle, Markdown+YAML records, Vitest, Codex-inspired tool lifecycle concepts, ForgeCode-inspired eval boundaries.

---

## Product Thesis

0.0.1 proved that physics memory can be loaded progressively through a model tool. 0.0.2 should make real research sessions compile-ready from the moment they happen.

The agent should not need to know the final physics graph before starting a topic. Instead:

```text
papers, web pages, derivation notes, code observations, git diffs, benchmarks, failures
-> typed research-ledger events
-> precompile proposals
-> research actions and obligations
-> checked capsules / graph edges / harness cases
```

The goal is a soft control system for theoretical physics research:

- the model may explore freely;
- every substantive claim, formula, derivation step, code mapping, or benchmark observation becomes a typed object;
- typed objects create obligations such as source, dimension, convention, known-limit, formula-code mapping, or benchmark checks;
- unresolved blocking obligations prevent validated promotion and should downgrade final answers to provisional status;
- failures become reusable negative memory and harness candidates.

## Runtime Boundary

Do not replace Kimi's existing primitive tools. AITP research actions are semantic orchestration over those tools.

```text
Layer 0: primitive runtime tools
  Read / Write / Edit / Bash / Web / MCP / Git / Background / Subagent

Layer 1: agent-native capability tools
  SkillTool / PhysicsMemory / ResearchLedger / ResearchAction / BenchmarkHarness

Layer 2: AITP semantic actions
  derive_step / check_dimension / check_convention / map_formula_to_code
  inspect_git_history / run_minimal_benchmark / propose_capsule

Layer 3: WorkFrame policy
  active domain, active topic, allowed context, exposed tools, blocking obligations
```

Research actions call primitive tools when needed, but each primitive call should be attributed to a semantic action or explicitly recorded as a raw-tool escape.

## Three Progressive Disclosure Layers

### Skills

Skills remain procedural memory. They tell the agent how to work.

### Physics Memory

Physics memory remains compiled semantic memory. It tells the agent what is known and under which assumptions.

### Research Ledger

Research ledger is the source-backed event layer. It records what actually happened in the research session before the material is trusted enough to become physics memory.

The three layers should share the same disclosure pattern:

```text
session startup scans indexes
-> model sees compact availability/reminder
-> model calls tools to expand only the needed details
-> loads and writes are recorded
```

## Research Ledger Schema Draft

Use Markdown with YAML frontmatter, mirroring physics capsules while keeping source events separate from trusted memory.

```markdown
---
id: event.fqhe-cs.2026-06-01.flux-quantization-note
type: derivation_scratch
topic: fqhe-cs-effective-theory
domain: topological-order
status: captured
source_refs:
  - paper:zhang-hansson-kivelson-1989
depends_on:
  - source_excerpt:fqhe-cs.flux-attachment
candidate_capsule_kind: DerivationStep
open_questions:
  - check flux quantum convention
  - check CS level normalization
related_objects:
  - formula:fqhe.flux-quantization
---

## Observation

The derivation uses one flux quantum per attached statistical flux unit...
```

Event statuses:

- `captured`
- `parsed`
- `linked`
- `compiled`
- `promoted`
- `rejected`

Event types:

- `source_excerpt`
- `derivation_scratch`
- `equation_candidate`
- `assumption_candidate`
- `code_observation`
- `git_diff_observation`
- `benchmark_observation`
- `failure_observation`
- `decision_note`
- `tool_run`

## ActionAlgebra Contract

A research action is a typed transition:

```ts
interface ResearchActionDefinition {
  id: string;
  phase: 'scope' | 'source' | 'explore' | 'derive' | 'validate' | 'code' | 'benchmark' | 'compile';
  inputKinds: readonly ResearchObjectKind[];
  outputKinds: readonly ResearchObjectKind[];
  preconditions: readonly ActionPrecondition[];
  effects: readonly ActionEffect[];
  generatedObligations: readonly ObligationTemplate[];
  validators: readonly ActionValidator[];
  primitiveToolPolicy: PrimitiveToolPolicy;
}
```

The action does not need to force a single reasoning route. It should make the consequences of a reasoning step explicit:

```text
derive_step
-> creates DerivationStepCandidate and ClaimCandidate
-> creates source, dimension, convention, known-limit obligations
-> writes ledger event
-> blocks validated promotion until required obligations pass
```

## WorkFrame Contract

`WorkFrame` is the session-local state for one active research problem.

```ts
interface WorkFrame {
  id: string;
  domain: PhysicsDomainId;
  topic: string;
  goal: string;
  activeObjectIds: readonly string[];
  assumptionIds: readonly string[];
  conventionIds: readonly string[];
  sourceRefs: readonly string[];
  openObligationIds: readonly string[];
  contextPackId?: string;
  trustState: 'exploratory' | 'deriving' | 'checking' | 'validated' | 'blocked';
}
```

Multiple work frames may exist in one session. A LibRPA feature-development frame and a FQHE theory-explanation frame must not share context unless a bridge is explicit.

## Obligation Contract

Obligations are the main convergence mechanism.

```ts
interface ResearchObligation {
  id: string;
  kind:
    | 'source_support'
    | 'dimension_check'
    | 'convention_check'
    | 'symbol_closure'
    | 'dependency_closure'
    | 'known_limit'
    | 'code_mapping'
    | 'benchmark'
    | 'human_decision';
  targetObjectId: string;
  severity: 'blocking' | 'important' | 'advisory';
  reason: string;
  requiredActionId: string;
  status: 'open' | 'passed' | 'failed' | 'waived';
}
```

Blocking obligations should prevent `validated` promotion and force final answers to disclose provisional status.

## Coordination With Kimi, Codex, And ForgeCode

### Kimi Code

Kimi remains the base runtime. Use its TypeScript agent loop, tool manager, skills, session lifecycle, records/replay, MCP, hooks, permissions, and compaction.

0.0.2 should integrate through:

- optional session managers, not global prompt patches;
- builtin tools registered behind experimental flags;
- `AgentRecords` append-only events;
- existing permission and tool exposure pathways.

### Codex

Borrow lifecycle structure:

- explicit action/tool source;
- stable pre-action and post-action payloads;
- primitive tool attribution inside semantic action execution;
- cancellation/error/outcome distinctions;
- patch/edit discipline and command lifecycle ideas where they fit Kimi.

### ForgeCode

Borrow harness boundaries:

- eval cases as durable files;
- action sequence plus validation contract;
- task data separated from runner;
- failures convertible into regression candidates.

## Tool Exposure Policy

The model should not always see every capability. Action exposure depends on active WorkFrame:

```text
FQHE/topological theory:
  expose PhysicsMemory, ResearchLedger, ResearchAction, Web/source tools if needed
  prefer derivation/convention/known-limit actions

LibRPA feature development:
  expose PhysicsMemory, ResearchLedger, ResearchAction, Git/Bash/Read/Edit, BenchmarkHarness, LibRPA MCP
  prefer formula-code mapping, git inspection, smoke benchmark, intermediate observable checks

Literature learning:
  expose source capture, citation tools, PhysicsMemory, ResearchLedger
  keep mutation and benchmark tools deferred
```

Raw primitive tools remain available when the runtime allows them, but unstructured usage should be recorded as:

```text
raw_tool_escape:
  reason: action missing / exploratory / emergency
  followup: propose ResearchAction or harness case
```
