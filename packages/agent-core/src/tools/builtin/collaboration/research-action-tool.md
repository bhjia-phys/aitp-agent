Inspect and record Hakimi research actions.

Use this tool to list available semantic research actions, render the primitive tool plan for an action, ask the scheduler which action should handle open obligations next, open/switch/list/close active WorkFrames, compile/load bounded ContextPacks for a WorkFrame, inspect the attached DomainPack manifest, list/load WorkFrame-scoped evidence refs, inspect AITP runtime payload profile metadata, inspect/search AITP curated heuristic RAG context, draft read-only curated RAG promotion plans and prefilled write-bridge call drafts, inspect handoff guard remediation taxonomy metadata, inspect AITP write-bridge handoff readiness, run deterministic in-process research executors, execute a configured AITP write bridge, or record the outcome of a semantic research action. This tool can execute only built-in graph queries, formalization-blueprint export, registered benchmark adapters, scoped ResearchLedger evidence inspection, read-only handoff remediation taxonomy/readiness inspection, read-only AITP runtime payload profile inspection, read-only AITP curated RAG inspection/search/draft planning, and explicitly configured AITP write-bridge operations. It does not execute arbitrary shell, git, web, MCP, or HPC work by itself. Primitive tool execution should remain attributed separately and then recorded back through the action result.

Curated RAG and record-ref repair handoffs remain draft-only until a separate explicit `execute_aitp_write_bridge` call provides top-level `aitp_operation` and `aitp_payload`. `inspect_aitp_write_bridge_handoff_readiness` can check the same handoff guard before execution while returning `bridge_called=false`. When an optional `aitp_handoff` fails guard verification, the result includes `handoff_guard_failure` and a `remediation_summary` with stable repair steps: `copy_missing_handoff_field_from_draft`, `align_explicit_execute_args_with_handoff_tool_call`, `redraft_handoff_or_restore_hash_input`, `redraft_or_resolve_blocking_diagnostics`, or `inspect_handoff_guard_failure`. `inspect_handoff_guard_remediation_taxonomy` returns the same failure-code to repair-step map as read-only metadata. These are repair hints only; they never mutate the handoff, call the bridge, record evidence, validate, satisfy final gates, or update claim trust.

Structured `promotion_carried_ref_handoffs` for curated RAG write-call drafts fail closed before suggestions or next-call pointers render when required fields are missing or inconsistent. The failure output includes `carried_ref_handoff_failure` with stable `code`, `field`, `path`, and `remediation_summary` metadata, while keeping `bridge_called=false`, `executes_write_now=false`, and no validation/trust authority. `list_actions` also renders `carried_ref_handoff_diagnostic_taxonomy` as read-only metadata so models can inspect the carried-ref failure-code map without intentionally submitting malformed handoffs.

When a ResearchContextPack includes `curated_rag_carried_ref_repair_sequence`, treat it as a host reminder for the safe repair order: inspect taxonomy metadata, prepare a fresh draft, apply explicit reviewed overrides, inspect readiness, and execute only through a separate explicit `execute_aitp_write_bridge` call. It is not permission to render suggestions, mutate payloads, call bridges, validate, satisfy final gates, or update claim trust.

When that repair sequence is paired with a concrete `carried_ref_handoff_failure` `code` and `path`, the ContextPack may include a `draft_aitp_curated_rag_write_bridge_call` repair binding. The binding is only a model-facing affordance: provide the chunk, selected promotion stage or operation, and reviewed overrides explicitly, then inspect readiness and execute through a separate explicit call. Do not infer payload values, validation, source support, final-gate satisfaction, or trust changes from the binding.

Carried-ref repair drafts may also render `carried_ref_repair_readiness_echo` around draft output, readiness inspection, and explicit-execute precheck. Treat it as progress visibility only: it can say reviewed overrides are still needed, readiness inspection is ready, readiness inspection passed, or explicit-execute precheck passed, but it never authorizes execution, mutates payloads, calls the bridge by itself, records validation/source support, satisfies final gates, or changes claim trust.

After a carried-ref repair path reaches a successful explicit AITP source asset, reference location, or evidence write result, `carried_ref_repair_result_summary` may connect the returned record/ref to the handoff id, readiness checklist, and repair hint operations that prepared it. Treat that as model-facing audit context only: it does not mutate the next payload, execute another write, validate source support, satisfy final gates, or change claim trust.
