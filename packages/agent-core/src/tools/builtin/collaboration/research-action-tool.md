Inspect and record Hakimi research actions.

Use this tool to list available semantic research actions, render the primitive tool plan for an action, ask the scheduler which action should handle open obligations next, open/switch/list/close active WorkFrames, compile/load bounded ContextPacks for a WorkFrame, inspect the attached DomainPack manifest, list/load WorkFrame-scoped evidence refs, inspect AITP runtime payload profile metadata, draft generic AITP write-bridge calls with local readiness diagnostics, inspect/search AITP curated heuristic RAG context, inspect AITP literature source-review handoff and comparison-draft packets, draft read-only curated RAG promotion plans and prefilled write-bridge call drafts, inspect handoff guard remediation taxonomy metadata, inspect AITP write-bridge handoff readiness, inspect source-context review handoff readiness, run deterministic in-process research executors, execute a configured AITP write bridge, or record the outcome of a semantic research action. This tool can execute only built-in graph queries, formalization-blueprint export, registered benchmark adapters, scoped ResearchLedger evidence inspection, read-only handoff remediation taxonomy/readiness inspection, read-only AITP runtime payload profile inspection, read-only generic AITP write-call drafting, read-only AITP curated RAG inspection/search/draft planning, read-only AITP literature source-review handoff and comparison-draft inspection, and explicitly configured AITP write-bridge operations. It does not execute arbitrary shell, git, web, MCP, or HPC work by itself. Primitive tool execution should remain attributed separately and then recorded back through the action result.

`draft_aitp_write_bridge_call` is the default write-planning surface for non-curated AITP writes such as `startResearchRun`, `recordResearchRunEvent`, and `recordExploratoryRecord`. It returns a reviewed payload draft, missing-field diagnostics, allowed enum values, a readiness inspection call, and a guarded `execute_aitp_write_bridge_handoff`. It never calls the bridge, records evidence, validates, satisfies final gates, or updates claim trust. Use it before explicit execution so models do not guess AITP payload schemas by trial-and-error.

Curated RAG and record-ref repair handoffs remain draft-only until a separate explicit `execute_aitp_write_bridge` call provides top-level `aitp_operation` and `aitp_payload`. `inspect_aitp_write_bridge_handoff_readiness` can check the same handoff guard before execution while returning `bridge_called=false`. When an optional `aitp_handoff` fails guard verification, the result includes `handoff_guard_failure` and a `remediation_summary` with stable repair steps: `copy_missing_handoff_field_from_draft`, `align_explicit_execute_args_with_handoff_tool_call`, `redraft_handoff_or_restore_hash_input`, `redraft_or_resolve_blocking_diagnostics`, or `inspect_handoff_guard_failure`. `inspect_handoff_guard_remediation_taxonomy` returns the same failure-code to repair-step map as read-only metadata. These are repair hints only; they never mutate the handoff, call the bridge, record evidence, validate, satisfy final gates, or update claim trust.

Structured `promotion_carried_ref_handoffs` for curated RAG write-call drafts fail closed before suggestions or next-call pointers render when required fields are missing or inconsistent. The failure output includes `carried_ref_handoff_failure` with stable `code`, `field`, `path`, and `remediation_summary` metadata, while keeping `bridge_called=false`, `executes_write_now=false`, and no validation/trust authority. `list_actions` also renders `carried_ref_handoff_diagnostic_taxonomy` as read-only metadata so models can inspect the carried-ref failure-code map without intentionally submitting malformed handoffs.

When a ResearchContextPack includes `curated_rag_carried_ref_repair_sequence`, treat it as a host reminder for the safe repair order: inspect taxonomy metadata, prepare a fresh draft, apply explicit reviewed overrides, inspect readiness, and execute only through a separate explicit `execute_aitp_write_bridge` call. It is not permission to render suggestions, mutate payloads, call bridges, validate, satisfy final gates, or update claim trust.

When that repair sequence is paired with a concrete `carried_ref_handoff_failure` `code` and `path`, the ContextPack may include a `draft_aitp_curated_rag_write_bridge_call` repair binding. The binding is only a model-facing affordance: provide the chunk, selected promotion stage or operation, and reviewed overrides explicitly, then inspect readiness and execute through a separate explicit call. Do not infer payload values, validation, source support, final-gate satisfaction, or trust changes from the binding.

Carried-ref repair drafts may also render `carried_ref_repair_readiness_echo` around draft output, readiness inspection, and explicit-execute precheck. Treat it as progress visibility only: it can say reviewed overrides are still needed, readiness inspection is ready, readiness inspection passed, or explicit-execute precheck passed, but it never authorizes execution, mutates payloads, calls the bridge by itself, records validation/source support, satisfies final gates, or changes claim trust.

After a carried-ref repair path reaches a successful explicit AITP source asset, reference location, or evidence write result, `carried_ref_repair_result_summary` may connect the returned record/ref to the handoff id, readiness checklist, and repair hint operations that prepared it. Treat that as model-facing audit context only: it does not mutate the next payload, execute another write, validate source support, satisfy final gates, or change claim trust.

If a later ContextPack includes `curated_rag_carried_ref_repair_result`, it may include a `source.review_context` binding before the continuation `draft_aitp_curated_rag_write_bridge_call` binding. Use the review binding to inspect source text, chunk scope, claim scope, and candidate support before choosing extraction, source-support validation, a fresh AITP draft, or a blocker. Use the returned canonical ref only as candidate reviewed override input for a fresh draft. Neither binding infers chunk, promotion stage, payload values, source support, validation, final-gate satisfaction, or trust changes, and the draft path still requires readiness inspection and a separate explicit execute call.

When finishing `source.review_context`, provide structured `action_output.kind="source_review_context"` with one decision: `extract`, `validate_check_source_support`, `fresh_aitp_draft`, or `blocker`. The output must include reviewed refs, rationale, matching `nextSuggestedActions`, and `nonEvidentiaryBoundary` flags with `recordsValidationResult=false`, `sourceSupportResult=false`, `claimTrustMutation="none"`, and `canUpdateClaimTrust=false`. `source_context_review_outcome` is routing metadata only; it does not prove source support, record validation, execute AITP writes, satisfy final gates, or update claim trust.

Before following a `source_context_review_outcome` binding, `inspect_source_context_review_handoff` can check that the ContextPack binding still has the expected adapter id, matching decision-to-next-action mapping, `allowedNextToolCall`, and no-trust/no-write boundary flags. The inspection returns `source_context_review_handoff_readiness` with `bridge_called=false`, `executes_write_now=false`, `records_validation_result=false`, `source_support_result=false`, and `claim_trust_mutation="none"`. Passing readiness only allows the named next ResearchAction to be planned explicitly; it does not execute that action, infer payload values, prove source support, record validation, satisfy final gates, or update claim trust.

`plan_primitive_tools` may also receive the same `context_pack_id` and `action_binding_id` for a `source_context_review_outcome` binding. In that mode it renders the normal primitive plan plus the bounded source-review handoff context, and fails closed if the binding action id does not match `action_id`.

`inspect_literature_source_review_handoff` calls the AITP-owned `literature_source_review_handoff` read surface through MCP-first transport with CLI fallback. The packet composes literature intake suggestion, record-ref lookup, source-stack coverage, and source reconstruction review context for choosing a later `source.review_context` plan. It is handoff context only: `bridge_called=false`, `executes_write_now=false`, `records_validation_result=false`, `source_support_result=false`, `write_executed=false`, and `claim_trust_mutation="none"`. Recommended next entrypoints are suggestions for explicit later AITP calls, not automatic payload mutation, evidence, validation, final-gate satisfaction, or trust authority.

`inspect_literature_comparison_draft` calls the AITP-owned `literature_comparison_draft` read surface through MCP-first transport with CLI fallback. The packet scaffolds a bounded source-set comparison question, source refs, dimensions, suggested agreement/disagreement/missing-evidence sections, and recommended next AITP entrypoints for an explicit later `source.compare_literature` plan. It is comparison-draft context only: `draft_creates_records=false`, `bridge_called=false`, `executes_write_now=false`, `records_validation_result=false`, `source_support_result=false`, `write_executed=false`, and `claim_trust_mutation="none"`. The draft is not a literature comparison record, evidence, validation, final-gate satisfaction, or trust authority.
